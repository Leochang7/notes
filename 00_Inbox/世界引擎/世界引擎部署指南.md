## 本地开发环境

### Docker Compose 部署

```yaml
# docker-compose.yml
version: '3.8'

services:
  # PostgreSQL
  postgres:
    image: postgres:15
    environment:
      POSTGRES_DB: world_engine
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: ${DB_PASSWORD}
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data

  # Redis
  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    volumes:
      - redis_data:/data

  # Milvus (向量数据库)
  etcd:
    image: quay.io/coreos/etcd:v3.5.5
    environment:
      ETCD_AUTO_COMPACTION_MODE: revision
      ETCD_AUTO_COMPACTION_RETENTION: 1000
      ETCD_QUOTA_BACKEND_BYTES: 4294967296
    volumes:
      - etcd_data:/etcd

  minio:
    image: minio/minio:RELEASE.2023-03-20T20-16-18Z
    environment:
      MINIO_ACCESS_KEY: minioadmin
      MINIO_SECRET_KEY: minioadmin
    command: minio server /minio_data
    volumes:
      - minio_data:/minio_data

  milvus:
    image: milvusdb/milvus:v2.3.3
    depends_on:
      - etcd
      - minio
    ports:
      - "19530:19530"
    environment:
      ETCD_ENDPOINTS: etcd:2379
      MINIO_ADDRESS: minio:9000

  # Kafka (消息队列)
  zookeeper:
    image: confluentinc/cp-zookeeper:7.5.0
    environment:
      ZOOKEEPER_CLIENT_PORT: 2181
      ZOOKEEPER_TICK_TIME: 2000

  kafka:
    image: confluentinc/cp-kafka:7.5.0
    depends_on:
      - zookeeper
    ports:
      - "9092:9092"
    environment:
      KAFKA_BROKER_ID: 1
      KAFKA_ZOOKEEPER_CONNECT: zookeeper:2181
      KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://kafka:9092
      KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 1

  # 控制节点
  control_node:
    build:
      context: ./control-node
      dockerfile: Dockerfile
    depends_on:
      - postgres
      - redis
      - kafka
      - milvus
    ports:
      - "8080:8080"
    environment:
      DB_URL: postgresql://postgres:${DB_PASSWORD}@postgres:5432/world_engine
      REDIS_URL: redis://redis:6379
      KAFKA_BROKERS: kafka:9092
      MILVUS_HOST: milvus
      MILVUS_PORT: 19530

  # 推理节点 (需要GPU)
  inference_node:
    build:
      context: ./inference-node
      dockerfile: Dockerfile
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: 1
              capabilities: [gpu]
    ports:
      - "8081:8081"
    environment:
      MODEL_PATH: /models
      CUDA_VISIBLE_DEVICES: 0

  # Prometheus (监控)
  prometheus:
    image: prom/prometheus:latest
    ports:
      - "9090:9090"
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml
      - prometheus_data:/prometheus

  # Grafana (可视化)
  grafana:
    image: grafana/grafana:latest
    ports:
      - "3000:3000"
    environment:
      GF_SECURITY_ADMIN_PASSWORD: admin
    volumes:
      - grafana_data:/var/lib/grafana

volumes:
  postgres_data:
  redis_data:
  etcd_data:
  minio_data:
  prometheus_data:
  grafana_data:
```

### 启动开发环境

```bash
# 1. 克隆仓库
git clone https://github.com/your-org/world-engine.git
cd world-engine

# 2. 配置环境变量
cp .env.example .env
# 编辑 .env 文件，填写密码等敏感信息

# 3. 启动服务
docker-compose up -d

# 4. 等待服务启动
docker-compose ps

# 5. 查看日志
docker-compose logs -f control_node

# 6. 运行测试
cargo test --workspace

# 7. 启动本地推理服务
# 如果有GPU，启动本地模型
./scripts/start_local_model.sh --model-path ./models/llama-2-7b

# 8. 构建世界
cargo run --bin world_builder -- --seed seeds/jianghu.txt

# 9. 启动仿真
cargo run --bin engine -- --config config/development.yaml
```

kubectl get pods -n world-engine

kubectl logs -f deployment/control-node -n world-engine

# 8. 扩容推理节点

### 告警规则

```yaml
# prometheus/alerts.yml
groups:
  - name: world-engine
    interval: 30s
    rules:
      - alert: HighErrorRate
        expr: rate(http_requests_total{status=~"5.."}[5m]) > 0.1
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "High error rate detected"

      - alert: HighLatency
        expr: histogram_quantile(0.95, http_request_duration_seconds) > 1.0
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "High latency detected"

      - alert: HighMemoryUsage
        expr: container_memory_usage_bytes / container_spec_memory_limit_bytes > 0.9
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "High memory usage"

      - alert: PodNotReady
        expr: kube_pod_status_ready{namespace="world-engine"} == 0
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "Pod not ready"
```

## 故障处理

### 常见问题

#### 1. Agent 响应延迟过高

**症状**：玩家与Agent互动时，响应时间超过1秒

**排查步骤**：

```bash
# 检查推理节点负载
kubectl top pods -l app=inference-node -n world-engine

# 检查LLM推理延迟
kubectl logs -l app=inference-node -n world-engine | grep "inference_duration"

# 检查语义缓存命中率
curl http://control-node:8080/metrics | grep cache_hit_rate
```

**解决方案**：

```yaml
# 临时扩容推理节点
kubectl scale deployment inference-node --replicas=10 -n world-engine

# 优化缓存配置
kubectl patch configmap world-engine-config -n world-engine -p '{
  "data": {
    "cache_size": "10000",
    "cache_ttl": "600"
  }
}'
```

#### 2. 区域转换卡顿

**症状**：玩家传送时出现明显延迟

**排查步骤**：

```bash
# 检查区域管理器日志
kubectl logs -l app=control-node -n world-engine | grep "hydration"

# 检查Agent数量
curl http://control-node:8080/api/regions | jq '.[].agent_count'

# 检查数据库查询性能
kubectl exec -it postgres-0 -n world-engine -- psql -c "EXPLAIN ANALYZE SELECT * FROM agents WHERE region_id = '...'"
```

**解决方案**：

```rust
// 预加载常用区域模板
impl RegionManager {
    async fn preload_regions(&self, region_ids: Vec<RegionId>) -> Result<()> {
        for region_id in region_ids {
            self.hydrator.prepare_template(region_id).await?;
        }
        Ok(())
    }
}
```

#### 3. 内存泄漏

**症状**：控制节点内存持续增长

**排查步骤**：

```bash
# 检查内存使用
kubectl top pods -l app=control-node -n world-engine

# 检查Agent数量
curl http://control-node:8080/api/agents | jq '.total_count'

# 检查未清理的记忆
kubectl exec -it postgres-0 -n world-engine -- psql -c "SELECT COUNT(*) FROM memories WHERE expires_at < NOW()"
```

**解决方案**：

```yaml
# 配置定期清理
apiVersion: batch/v1
kind: CronJob
metadata:
  name: memory-cleanup
  namespace: world-engine
spec:
  schedule: "0 * * * *"  # 每小时
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: cleanup
            image: world-engine/cleanup:v1.0.0
            command: ["/usr/bin/cleanup-memories"]
```

### 应急处理流程

```yaml
# 1. 故障检测
# 监控告警触发 -> 自动创建工单 -> 通知运维人员

# 2. 故障定位

# 3. 故障恢复
# - 轻微故障：自动恢复（重试、降级、熔断）
# - 中等故障：人工干预（扩容、配置调整）
# - 严重故障：紧急预案（回滚、限流、关停非核心服务）

# 4. 故障复盘
# - 记录故障处理过程
# - 分析根本原因
# - 制定改进措施
# - 更新应急预案
```
